# 1. Mô tả ngắn gọn ý đồ thiết kế quy trình 3 bước

Bài toán cần xây dựng tính năng **Rate Limiting** cho API Gateway của ứng dụng tài chính **SafePay**. Mỗi API Key chỉ được phép gửi tối đa **100 requests trong 1 phút**. Nếu vượt giới hạn, hệ thống trả về HTTP **429 Too Many Requests**.

Em thiết kế quy trình làm việc với AI theo 3 bước tăng dần:

```text
Bước 1: Hiểu và chọn thuật toán
→ Yêu cầu AI phân tích nhiều phương án như Fixed Window, Token Bucket, Sliding Window.

Bước 2: Phân tích bối cảnh hệ thống phân tán
→ Yêu cầu AI chỉ ra vì sao lưu counter trong local memory sẽ sai khi chạy nhiều server, sau đó đề xuất Redis làm kho lưu trữ tập trung.

Bước 3: Sinh mã nguồn có kiểm soát
→ Yêu cầu AI viết code Java Spring Boot dùng Redis, có xử lý fallback khi Redis lỗi, ghi log bằng @Slf4j và trả về HTTP 429 khi vượt giới hạn.
```

Cách chia 3 bước này giúp AI không nhảy ngay vào viết code, mà đi từ **phân tích thuật toán → phân tích kiến trúc phân tán → sinh mã nguồn production-ready**.

````markdown
# Phân tích thuật toán Rate Limiting cho SafePay

## 1. Fixed Window

### Cách hoạt động

Fixed Window chia thời gian thành các cửa sổ cố định, ví dụ mỗi cửa sổ là 1 phút.

Với quy tắc của SafePay:

- Mỗi API Key có một counter riêng.
- Trong mỗi phút, nếu counter <= 100 thì cho phép request.
- Nếu counter > 100 thì chặn và trả HTTP 429.
- Khi sang phút mới, counter được reset về 0.

Ví dụ:

```text
rate-limit:API_KEY_A:2026-06-22-10:01 = 100
```
````

### Ưu điểm

- Dễ hiểu.
- Dễ triển khai.
- Hiệu năng tốt.
- Phù hợp với yêu cầu đơn giản như 100 requests/phút.
- Dễ dùng với Redis bằng INCR và EXPIRE.

### Nhược điểm

- Có thể bị burst ở ranh giới thời gian.
- Ví dụ client gửi 100 request ở giây thứ 59 và 100 request ở giây thứ 60, tổng cộng 200 request trong thời gian rất ngắn nhưng vẫn có thể được chấp nhận.
- Độ chính xác không cao bằng Sliding Window.

### Độ phức tạp Java

Thấp. Có thể triển khai nhanh bằng Spring Boot Filter hoặc Interceptor.

### Mức độ phù hợp

Phù hợp cho giai đoạn đầu vì SafePay cần một giải pháp đơn giản, dễ kiểm soát và dễ vận hành.

---

## 2. Token Bucket

### Cách hoạt động

Mỗi API Key có một bucket chứa token. Mỗi request tiêu thụ 1 token. Hệ thống nạp lại token theo tốc độ cố định.

Ví dụ:

- Bucket tối đa 100 token.
- Mỗi phút nạp lại 100 token.
- Nếu còn token thì cho phép request.
- Nếu hết token thì trả HTTP 429.

### Ưu điểm

- Xử lý burst tốt hơn Fixed Window.
- Linh hoạt hơn.
- Phù hợp với hệ thống cần kiểm soát traffic mềm mại hơn.
- Có thể cho phép người dùng gửi nhanh một số request trong ngắn hạn nếu bucket còn token.

### Nhược điểm

- Triển khai phức tạp hơn Fixed Window.
- Cần lưu thêm trạng thái:
  - Số token còn lại.
  - Thời điểm refill gần nhất.

- Nếu triển khai phân tán cần đảm bảo atomic update, thường phải dùng Lua Script trong Redis.

### Độ phức tạp Java

Trung bình đến cao. Cần cẩn thận khi tính toán token và đồng bộ dữ liệu.

### Mức độ phù hợp

Phù hợp với hệ thống production có yêu cầu kiểm soát traffic mượt hơn, nhưng có thể hơi phức tạp cho giai đoạn đầu.

---

## 3. Sliding Window

### Cách hoạt động

Sliding Window đếm số request trong một khoảng thời gian trượt gần nhất, ví dụ 60 giây gần nhất tính từ thời điểm hiện tại.

Thay vì reset theo phút cố định, thuật toán kiểm tra chính xác số request trong 60 giây vừa qua.

### Ưu điểm

- Chính xác hơn Fixed Window.
- Giảm lỗi burst ở ranh giới thời gian.
- Phù hợp với hệ thống tài chính cần kiểm soát chặt.

### Nhược điểm

- Phức tạp hơn.
- Cần lưu timestamp của nhiều request.
- Có thể tốn bộ nhớ nếu traffic lớn.
- Khi dùng Redis thường cần Sorted Set hoặc Lua Script.

### Độ phức tạp Java

Cao hơn Fixed Window. Cần xử lý timestamp, xóa record cũ và đếm request trong window.

### Mức độ phù hợp

Phù hợp nếu SafePay yêu cầu độ chính xác cao, nhưng chưa cần thiết cho giai đoạn đầu nếu mục tiêu là triển khai nhanh và ổn định.

### AI Response

# Thiết kế Rate Limiter phân tán với Redis cho SafePay

## 1. Vì sao local memory không phù hợp?

Trong hệ thống chỉ có một server, việc lưu counter trong memory có thể hoạt động tạm được.

Tuy nhiên, SafePay chạy trên cụm 3 API Gateway instance:

```text
Client
  |
Load Balancer
  |
  |---- API Gateway 1
  |---- API Gateway 2
  |---- API Gateway 3
```

Nếu mỗi server tự lưu counter riêng, dữ liệu sẽ không đồng nhất.

Ví dụ:

```text
API Key = abc123

Gateway 1 đếm: 100 requests
Gateway 2 đếm: 100 requests
Gateway 3 đếm: 100 requests
```

Từng gateway đều nghĩ API Key chưa vượt quá 100 request/phút, nhưng thực tế tổng số request là 300 request/phút.

Vì vậy, local memory không phù hợp vì:

- Không chia sẻ trạng thái giữa các server.
- Không đảm bảo giới hạn toàn cục.
- Khi server restart, counter bị mất.
- Khi scale thêm server, giới hạn càng sai lệch.
- Không phù hợp với hệ thống tài chính yêu cầu kiểm soát chặt.

---

## 2. Kiến trúc Redis tập trung

Redis được dùng làm kho lưu counter tập trung.

```text
Client
  |
  v
Load Balancer
  |
  v
API Gateway Cluster
  |---- Gateway 1
  |---- Gateway 2
  |---- Gateway 3
  |
  v
Redis Central Store
```

Tất cả Gateway đều đọc và ghi counter vào cùng một Redis.

Redis key đề xuất:

```text
rate-limit:{apiKey}
```

Ví dụ:

```text
rate-limit:abc123 = 57
TTL = 60 seconds
```

---

## 3. Luồng xử lý request

### Bước 1: Client gửi request

Client gửi request đến API Gateway kèm header:

```text
X-API-KEY: abc123
```

### Bước 2: Gateway tạo Redis key

Gateway tạo key:

```text
rate-limit:abc123
```

### Bước 3: Gateway tăng counter

Gateway gọi Redis:

```text
INCR rate-limit:abc123
```

Nếu giá trị trả về là 1, nghĩa là đây là request đầu tiên trong window hiện tại, Gateway set TTL:

```text
EXPIRE rate-limit:abc123 60
```

### Bước 4: Kiểm tra giới hạn

Nếu counter <= 100:

```text
Cho request đi tiếp
```

Nếu counter > 100:

```text
Trả HTTP 429 Too Many Requests
```

---

## 4. Các vấn đề cần chú ý

## 4.1. Atomic increment

Redis `INCR` là atomic. Điều này giúp tránh race condition khi nhiều Gateway cùng tăng counter cho cùng một API Key.

Ví dụ cùng lúc 10 request vào 3 Gateway, Redis vẫn tăng counter chính xác theo thứ tự.

---

## 4.2. TTL 60 giây

TTL giúp Redis tự xóa counter sau 60 giây.

Nếu không set TTL, Redis sẽ giữ key mãi mãi, gây rò rỉ bộ nhớ.

Luồng cơ bản:

```text
counter = INCR key
if counter == 1:
    EXPIRE key 60
```

Cần chú ý trường hợp `INCR` thành công nhưng `EXPIRE` thất bại. Trong hệ thống production, có thể dùng Lua Script để đảm bảo INCR và EXPIRE nằm trong cùng một thao tác logic.

---

## 4.3. Race condition

Nếu chỉ dùng local memory, race condition rất dễ xảy ra khi nhiều thread cùng cập nhật biến đếm.

Với Redis, `INCR` đã atomic nên giảm rủi ro. Tuy nhiên, cặp lệnh `INCR` và `EXPIRE` vẫn nên được đóng gói bằng Lua Script nếu yêu cầu chặt chẽ.

---

## 4.4. Redis downtime

Nếu Redis sập, Gateway không thể kiểm tra rate limit tập trung.

Có 2 chiến lược fallback:

### Fail-open

Cho request đi tiếp khi Redis lỗi.

Ưu điểm:

- Không làm gián đoạn người dùng thật.
- Phù hợp hệ thống ưu tiên availability.

Nhược điểm:

- Có thể bị abuse nếu Redis downtime.

### Fail-closed

Chặn request khi Redis lỗi.

Ưu điểm:

- Bảo vệ hệ thống tốt hơn.

Nhược điểm:

- Có thể chặn cả người dùng hợp lệ.

Với ứng dụng tài chính SafePay, nên chọn phương án cân bằng:

```text
Redis lỗi → dùng local fallback limiter tạm thời + log cảnh báo + alert monitoring
```

## 5. Redis Rate Limiter Service

```java
package com.safepay.gateway.ratelimit;

import com.safepay.gateway.config.RateLimitProperties;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.data.redis.RedisConnectionFailureException;
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.stereotype.Service;

import java.time.Duration;

@Slf4j
@Service
@RequiredArgsConstructor
public class RedisRateLimiterService {

    private final StringRedisTemplate redisTemplate;
    private final RateLimitProperties properties;

    public boolean isAllowed(String apiKey) {
        String redisKey = "rate-limit:" + apiKey;

        try {
            Long currentCount = redisTemplate.opsForValue().increment(redisKey);

            if (currentCount == null) {
                log.warn("Redis returned null counter for apiKey={}", maskApiKey(apiKey));
                throw new IllegalStateException("Redis counter is null");
            }

            if (currentCount == 1) {
                redisTemplate.expire(redisKey, Duration.ofSeconds(properties.getWindowSeconds()));
            }

            if (currentCount > properties.getMaxRequests()) {
                log.warn(
                        "Rate limit exceeded. apiKey={}, currentCount={}, limit={}",
                        maskApiKey(apiKey),
                        currentCount,
                        properties.getMaxRequests()
                );
                return false;
            }

            log.info(
                    "Request allowed by Redis rate limiter. apiKey={}, currentCount={}",
                    maskApiKey(apiKey),
                    currentCount
            );

            return true;
        } catch (RedisConnectionFailureException ex) {
            log.error("Redis connection failed. Switching to local fallback limiter. apiKey={}", maskApiKey(apiKey), ex);
            throw ex;
        } catch (Exception ex) {
            log.error("Unexpected Redis rate limiter error. Switching to local fallback limiter. apiKey={}", maskApiKey(apiKey), ex);
            throw ex;
        }
    }

    private String maskApiKey(String apiKey) {
        if (apiKey == null || apiKey.length() <= 4) {
            return "****";
        }
        return "****" + apiKey.substring(apiKey.length() - 4);
    }
}
```

## 6. Local Fallback Rate Limiter

```java
package com.safepay.gateway.ratelimit;

import com.safepay.gateway.config.RateLimitProperties;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Component;

import java.time.Instant;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.AtomicLong;

@Slf4j
@Component
@RequiredArgsConstructor
public class LocalFallbackRateLimiter {

    private final RateLimitProperties properties;

    private final Map<String, LocalCounter> counters = new ConcurrentHashMap<>();

    public boolean isAllowed(String apiKey) {
        long now = Instant.now().getEpochSecond();

        LocalCounter counter = counters.compute(apiKey, (key, existingCounter) -> {
            if (existingCounter == null || now >= existingCounter.windowStart + properties.getWindowSeconds()) {
                return new LocalCounter(now, new AtomicLong(1));
            }

            existingCounter.count.incrementAndGet();
            return existingCounter;
        });

        long currentCount = counter.count.get();

        if (currentCount > properties.getMaxRequests()) {
            log.warn(
                    "Rate limit exceeded by local fallback. apiKey={}, currentCount={}, limit={}",
                    maskApiKey(apiKey),
                    currentCount,
                    properties.getMaxRequests()
            );
            return false;
        }

        log.warn(
                "Request allowed by LOCAL FALLBACK limiter. This is temporary and not globally accurate. apiKey={}, currentCount={}",
                maskApiKey(apiKey),
                currentCount
        );

        return true;
    }

    private String maskApiKey(String apiKey) {
        if (apiKey == null || apiKey.length() <= 4) {
            return "****";
        }
        return "****" + apiKey.substring(apiKey.length() - 4);
    }

    private record LocalCounter(
            long windowStart,
            AtomicLong count
    ) {
    }
}
```

## 7. RateLimitFilter

```java
package com.safepay.gateway.filter;

import com.safepay.gateway.ratelimit.LocalFallbackRateLimiter;
import com.safepay.gateway.ratelimit.RedisRateLimiterService;
import jakarta.servlet.FilterChain;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.data.redis.RedisConnectionFailureException;
import org.springframework.stereotype.Component;
import org.springframework.web.filter.OncePerRequestFilter;

import java.io.IOException;

@Slf4j
@Component
@RequiredArgsConstructor
public class RateLimitFilter extends OncePerRequestFilter {

    private static final String API_KEY_HEADER = "X-API-KEY";

    private final RedisRateLimiterService redisRateLimiterService;
    private final LocalFallbackRateLimiter localFallbackRateLimiter;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain
    ) throws ServletException, IOException {

        String apiKey = request.getHeader(API_KEY_HEADER);

        if (apiKey == null || apiKey.isBlank()) {
            log.warn("Missing API Key in request");
            response.setStatus(HttpServletResponse.SC_UNAUTHORIZED);
            response.setContentType("application/json");
            response.getWriter().write("""
                    {
                      "status": 401,
                      "error": "Unauthorized",
                      "message": "Missing X-API-KEY header"
                    }
                    """);
            return;
        }

        boolean allowed;

        try {
            allowed = redisRateLimiterService.isAllowed(apiKey);
        } catch (RedisConnectionFailureException ex) {
            log.error("Redis unavailable. Using local fallback rate limiter.");
            allowed = localFallbackRateLimiter.isAllowed(apiKey);
        } catch (Exception ex) {
            log.error("Rate limiter error. Using local fallback rate limiter.");
            allowed = localFallbackRateLimiter.isAllowed(apiKey);
        }

        if (!allowed) {
            response.setStatus(429);
            response.setContentType("application/json");
            response.getWriter().write("""
                    {
                      "status": 429,
                      "error": "Too Many Requests",
                      "message": "Rate limit exceeded. Maximum 100 requests per minute."
                    }
                    """);
            return;
        }

        filterChain.doFilter(request, response);
    }
}
```

## 9. Đánh giá fallback khi Redis sập

Khi Redis hoạt động bình thường:

```text
RateLimitFilter → RedisRateLimiterService → Redis
```

Khi Redis lỗi:

```text
RateLimitFilter → LocalFallbackRateLimiter
```

Ưu điểm của fallback:

- Ứng dụng không bị chết hoàn toàn.
- Vẫn có lớp bảo vệ cơ bản.
- Có log cảnh báo rõ ràng.

Nhược điểm:

- Không chính xác toàn cụm 3 server.
- Mỗi server sẽ có bộ đếm riêng.
- Chỉ nên dùng tạm thời.

Trong production, cần thêm:

- Redis Sentinel hoặc Redis Cluster.
- Monitoring Redis.
- Alert khi fallback được kích hoạt.
- Circuit breaker nếu Redis lỗi liên tục.
